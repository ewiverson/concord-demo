Model Fitting Workflow

Design specification for the IC-CAP-inspired fitting UI and the model file format

Status: Design Document This page specifies the planned UX overhaul and model file format. Nothing described here is implemented yet. Implementation order: (1) model file format + save/load, (2) server routes, (3) tuner UI, (4) optimizer UI, (5) LLM bridge.

Motivation: The IC-CAP Analogy

Keysight IC-CAP is a device modeling platform widely used in semiconductor characterization. Its key insight is that the model developer, not the tool developer, defines the UI. A device engineer ships an MDL (Model Definition Language) file that contains in one place: the circuit equations, parameter bounds, measurement setups, data-processing transforms, and which plots to show during fitting. The platform provides infrastructure (instrument control, optimization, visualization); the MDL file drives what that infrastructure does.

Concord adopts the same philosophy for neural mass models. The platform provides: data loading, simulation runner, tuner, optimizer, and LLM bridge. The model file drives: which tabs to show, which parameters are tunable, what the fitting targets are, and how to measure fit quality. A user can start from a template, fit their data without touching any code, or extend the model equations and have those extensions immediately available in the fitting UI.

The Model File (.concord.py)

A model file is a plain Python module with a conventional structure. It is self-contained: everything needed to simulate, tune, and fit lives in one file. 

wendling_hup117.concord.py
├── class MyWendling(Model):     # the equations — user-editable
│     default_parameters()       # ParameterVector with names, values, bounds, units
│     simulate()                 # returns ModelOutput
│
├── LAYOUT = { ... }             # tab layout manifest — which plots, which params
├── TRANSFORMS = { ... }         # named fitting targets (Python callables)
└── FIT_CONFIG = { ... }         # algorithm, goal weights, optimizer kwargs

Being plain Python means:

  • Diffs are readable and version-controllable with git
  • An LLM can read it in full and produce coherent edits
  • The model class is the same Model ABC used by all concord-model-* packages — no new interface required
  • Transforms are just functions; users write standard numpy/scipy code

Templates vs. Project Files

Each concord-model-* package ships a templates/ directory with one or more .concord.py template files. Templates are read-only and are never overwritten by the tool. When a user starts a new project, the server copies the chosen template into their project directory — the user works on that copy.

Rule: Never Overwrite a Template User saves always write to the project copy (~/concord-projects/<project>/). Template files remain unchanged, always available as a known-good starting point.

Template Workflow

User opens concord-server → "Model Fitting" tab
  → "New project" dialog
      → picks a template (lists all installed concord-model-* templates)
      → enters a project name
  → Server copies template → ~/concord-projects/<project>/wendling.concord.py
  → Session binds to that project file
  → User loads BIDS recording
  → Tunes, fits, saves
  → Project file updated in-place (never the template)

A save captures three things together: any edits to the model class, the current best-fit ParameterVector, and any changes to TRANSFORMS or FIT_CONFIG. The file is always a complete, runnable Python module — not a patch on top of the template.

LAYOUT is a dict that tells the platform how to build the fitting UI for this model: which tabs to show, what to render on each tab, and which parameters are exposed in the tuner.

Transforms (TRANSFORMS)

Transforms are the user-extensible fitting targets — the Concord equivalent of IC-CAP's PEL/Python extraction transforms. Each is a Python callable that maps a Recording or ModelOutput to a scalar or 1-D array. The optimizer minimizes the distance between transform(real_data) and transform(simulated_output). 

import numpy as np
from concord_metrics_spectral import WelchPSD
from concord_metrics_univariate import BandPower


def baseline_psd_shape(data, params=None):
    """Log PSD 1–40 Hz — match shape, not absolute power."""
    result = WelchPSD().compute(data, params)
    freqs = result.metadata["freqs"]
    mask = (freqs >= 1) & (freqs <= 40)
    return np.log10(result.data[:, mask].mean(axis=0))  # (n_freqs,)


def seizure_band_power(data, params=None):
    """Gamma / delta band power ratio during the ictal epoch."""
    gamma = BandPower(fmin=30, fmax=80).compute(data, params).data.mean()
    delta = BandPower(fmin=1,  fmax=4 ).compute(data, params).data.mean()
    return gamma / (delta + 1e-9)                        # scalar

Key Design Insight Transforms decouple "what I'm trying to match" from the optimizer. The optimizer does not need to know anything about PSD shapes or seizure dynamics — it just minimizes distances between transform outputs. Model developers write transforms; the platform runs them.

This mirrors IC-CAP's separation of measurement data (MDM files) from extraction algorithms (PEL/Python transforms): the measurement and the extraction recipe are independent, so you can change one without touching the other.

Fit Config (FIT_CONFIG)

FIT_CONFIG describes the default optimization setup: which parameters to fit, which transforms to use as goals, how to weight them, and which algorithm to use. All of these can be overridden from the optimizer UI without editing the file. 

FIT_CONFIG = {
    "algorithm": "differential_evolution",   # nelder_mead | cmaes | bayesian
    "params_to_fit": ["A", "B", "G", "p", "sigma"],
    "goals": {
        "baseline_psd_shape": {
            "weight": 1.0,
            "metric": "mse",      # mse | correlation | dtw | custom callable
        },
        "seizure_band_power": {
            "weight": 2.0,
            "metric": "mse",
        },
        "onset_latency": {
            "weight": 0.5,
            "metric": "mse",
        },
    },
    "optimizer_kwargs": {
        "maxiter": 500,
        "tol":     1e-4,
        "workers": 4,
    },
}

concord-fit is the backend engine that consumes TRANSFORMS and FIT_CONFIG. It is exposed to the UI through server routes (see below) and never hard-imports specific model or metric packages — it discovers them through the entry points registry.

Tuner UI

The tuner is a persistent panel (collapsible sidebar) visible on any fitting tab. It provides immediate interactive feedback without needing to run the full optimizer.

  • One slider per parameter listed in LAYOUT["tuner"]["params"], labeled with display name and unit from ParameterVector
  • Lock checkbox per parameter — locked params are excluded from the current optimizer run but still show their value
  • Live re-simulation triggered on slider release (not continuous drag) — simulates the model and re-renders all plots on the current tab with real vs. simulated overlaid
  • "Snap to best" button — resets all sliders to the current best-fit ParameterVector from the most recent optimizer run
  • Numeric input field beside each slider for precise value entry
  • Bounds shown as range markers on the slider track

Optimizer UI

The optimizer panel drives concord-fit from the browser.

  • Algorithm dropdown — overrides FIT_CONFIG["algorithm"] for this run
  • Goal weight sliders — one per entry in FIT_CONFIG["goals"], live-editable without restarting a run
  • Freeze param checkboxes — subset of params_to_fit to include in this run (all checked by default)
  • Start / Stop / Pause controls — pause freezes parameter updates but keeps the best result so far; resume continues from that point
  • Convergence plot — objective value vs. iteration, live-updating
  • Parameter trajectory — each fitted param's value over iterations, useful for spotting parameters that are stuck at a bound
  • On completion: best parameters shown; user clicks "Accept" to push them to the tuner sliders (does not save the file automatically)

LLM Integration

IC-CAP requires the model developer to write PEL or Python extraction code by hand. Concord can lower this barrier: the model file is just Python, so an LLM can read it and produce coherent edits.

What the LLM Can Do

CapabilityExample
Model editing "Add a parameter for 1/f noise — update the equations, ParameterVector, LAYOUT, TRANSFORMS, and FIT_CONFIG"
Transform generation "Write a transform that measures the slope of the log PSD in the 1–10 Hz range"
Fitting guidance After a run: "Parameter B is at its lower bound and loss isn't converging — suggest widening bounds or changing the goal weighting"
Regime annotation "Looking at the loaded data, which Wendling regime does it resemble? What starting parameter values would you suggest?"

Interaction Pattern

  • Chat panel — collapsible sidebar, always available
  • Automatic context — the LLM receives the full current model file text, current parameter values, and (if a fit has run) the convergence history and residuals
  • Proposed edits as inline diff — when the LLM suggests code changes, they appear as a unified diff in a code review panel; user clicks Accept or Dismiss
  • No auto-execution — LLM output is never applied automatically; the same Accept/Dismiss flow applies regardless of how the edit was generated
Security Boundary LLM-generated code runs in the same Python process as the server. Users must explicitly click "Accept" before any code edit is written to disk. The server never executes a new model class until the file has been saved and explicitly reloaded by the user.

Server-Side Architecture

The fitting workflow requires new routes in concord-server and new server state. The heavy computation (simulation, optimization) stays in concord-fit; the server is a thin coordination layer.

New Routes

RoutePurpose
GET /api/model/listList available templates from installed concord-model-* packages
POST /api/model/loadLoad a .concord.py file; return LAYOUT, param names/bounds, FIT_CONFIG
POST /api/model/simulateRun model.simulate() with given ParameterVector; return timeseries JSON
POST /api/model/saveWrite the project .concord.py file (never a template path)
POST /api/fit/startLaunch a background fitting job
GET /api/fit/statusPoll fit progress: iteration count, current objective, current params
POST /api/fit/stopCancel the running fit; return best params found so far
POST /api/llm/chatSend prompt + context to Claude API; stream response back
POST /api/llm/applyWrite an LLM-proposed model file edit after user accepts

Out of Scope

The following are explicitly deferred and should not influence the initial implementation:

  • concord-connectome integration — connectivity optimizer tab requires parcellation and contact-to-region mapping, which belongs in a separate package
  • Stimulation simulation tab — driving a fitted model with a stimulus waveform is a later phase
  • Multi-node network models — the current Model ABC supports multi-node output but network connectivity is not yet wired to the fitting workflow
  • Statistical/population fitting — fitting across multiple subjects or sessions is a future concern
